# Chapter 2 — Background

## 2.1 Scope of this chapter

This chapter establishes the technical and regulatory ground on which the protocol proposed in Chapter 3 is built. Four bodies of work are relevant: the W3C PROV family of standards, which defines the general vocabulary for provenance on the web; PROV-AGENT, the closest prior work and the immediate predecessor of this work; C2PA, an industry standard for content-level provenance that is sometimes confused with pipeline-level provenance; and the EU AI Act, the regulatory instrument whose obligations the protocol is designed to discharge.

Two adjacent areas — provenance for ML training pipelines (PROV-ML, ProvONE) and FAIR data initiatives (FAIR4ML) — are mentioned briefly where they help locate the contribution, but they are not load-bearing for the argument.

The chapter ends with a synthesis that identifies what existing standards leave uncovered. This sets up the protocol design in Chapter 3.

---

## 2.2 W3C PROV — the vocabulary layer

The W3C PROV family (PROV-DM as the data model, PROV-O as the OWL ontology, and several supporting documents) is the de facto vocabulary for expressing provenance on the web. Its purpose is deliberately narrow: to provide a shared way of describing the origin and derivation of things, without committing to any domain-specific schema.

PROV is built on three primitives:

- **Entity** — a physical, digital, or conceptual thing whose provenance is being described. A document, a dataset, an LLM response, a model weight file are all entities.
- **Activity** — a dynamic process that occurs over a period of time and acts upon or generates entities. A model training run, an inference call, a database query are all activities.
- **Agent** — a person, software system, or organisation that bears some form of responsibility for an activity or for an entity's existence.

Around these primitives, PROV defines a small set of relationships: `wasGeneratedBy` (an entity was produced by an activity), `used` (an activity consumed an entity), `wasAssociatedWith` (an agent participated in an activity), `wasAttributedTo` (an entity is attributed to an agent), `wasDerivedFrom` (an entity was transformed from another), and `actedOnBehalfOf` (delegation between agents).

PROV also supports *qualified* relationships, which lift binary edges into objects that can themselves carry attributes — roles, timestamps, plans, or arbitrary domain extensions. Qualification is the mechanism by which downstream standards add specificity without breaking interoperability with the base vocabulary.

The strength of PROV is also its limitation. It tells us *that* an LLM response was generated by an inference activity associated with an agent on behalf of a person, but it does not, on its own, tell us anything about the model used, the prompt content, the latency, the temperature, or whether the response passed through a human reviewer before reaching a user. Domain-specific extensions are required for any of those facts to be expressible.

For the purposes of this work, PROV is the base vocabulary that both PROV-AGENT and the protocol proposed here build on. It is necessary but insufficient, and the contribution of this work is at the extension layer, not at the vocabulary layer.

## 2.3 Provenance for AI agents

### 2.3.1 PROV-AGENT

PROV-AGENT (Souza et al., IEEE e-Science 2025) is the closest published predecessor to this work and the reference point against which the contribution must be defined. It extends W3C PROV with classes and relationships specific to LLM agentic workflows.

PROV-AGENT introduces an `AIAgent` subclass of PROV's `Agent`, an `AIModelInvocation` subclass of `Activity` representing a single LLM call, an `AgentTool` subclass representing tool/function execution, and a hierarchy of activity subclasses (`Campaign`, `Workflow`, `Task`) modelling pipeline structure. On the entity side it adds `Prompt`, `ResponseData`, `AIModel`, and `DomainData` for tool inputs and outputs.

For each agent interaction it captures the prompt and response text, the model name and provider, the temperature and other inference parameters, execution timestamps and response latency, tool inputs and outputs, the agent identity (via `wasAttributedTo`), and inter-record linkage (via `wasInformedBy`). The implementation is a generic Python wrapper compatible with CrewAI, LangChain, and direct OpenAI calls. It has been deployed in an autonomous additive-manufacturing workflow at Oak Ridge National Laboratory.

PROV-AGENT covers, comprehensively, the *automated* portion of a multi-agent pipeline. An audit of a PROV-AGENT bundle can answer: what model produced this output, with what prompt, at what time, in what pipeline, in response to what tool call. For a scientific HPC workflow whose primary requirement is reproducibility — re-running the pipeline and obtaining the same result — this is sufficient.

The gaps relative to the requirements of this protocol are detailed in §2.6. Briefly: PROV-AGENT does not model human oversight events, does not map to any regulatory framework, stores raw prompts and responses by default rather than hashes, and provides no integrity seal over the record set.

### 2.3.2 Other AI/ML provenance work

A handful of other standards address adjacent problems. **PROV-ML** (Pimentel et al.) extends PROV to ML experiment tracking, focusing on training artefacts — datasets, hyperparameters, model versions, evaluation metrics. It is centred on the model-development lifecycle, not the deployed-inference lifecycle, and has no concept of an interactive agent or a human reviewer.

**ProvONE** extends PROV with constructs for scientific workflow systems (Kepler, Taverna, VisTrails). Its concerns — workflow templates, parameter sweeps, sub-workflow nesting — predate the LLM-agent paradigm and do not map cleanly onto it.

**PROV-DfA** (Provenance for Data-from-Activities, in some literatures rendered as Data-flow Analysis) addresses human-steered scientific workflows where a human selects parameters or directs branching. It models human *actions* as activities, not human *oversight of AI output*. The distinction matters: in PROV-DfA the human is the actor whose work is being recorded; in the EU AI Act framing, the AI is the actor and the human is the supervisor whose approval, rejection, or override constitutes regulatory evidence.

**FAIR4ML** applies the FAIR principles (Findable, Accessible, Interoperable, Reusable) to ML model artefacts. It is a metadata schema for model registries, not an execution-trace standard.

None of these standards provide a model for human oversight of an LLM agent's output, and none map to any regulatory regime.

## 2.4 C2PA — content authenticity, not pipeline provenance

The Coalition for Content Provenance and Authenticity (C2PA) maintains a specification for embedding cryptographically signed provenance metadata directly into media files — images, video, audio, documents. Its primary use case is combating misinformation: a news photograph carries a manifest signed by the camera (or the editing tool, or the publisher) attesting to where it came from and how it was modified.

C2PA is an *artifact-level* provenance standard. The unit of provenance is a single content object, and the claims attached to that object describe its history of capture and edit. Multiple manifests may be chained when a file passes through successive tools, with each tool appending and signing its own claim.

For LLM pipelines, C2PA is relevant in one specific direction. The EU AI Act, Article 50(2), requires that AI-generated content (audio, image, video, text) be machine-readably marked as artificially generated. C2PA's content credentials provide one viable mechanism for that marker. But the marker lives on the artefact, not on the pipeline that produced it, and the C2PA manifest does not capture which agents executed in what order, which tools were called, or whether a human reviewer approved the output.

The protocol proposed here is a *pipeline-level* provenance standard. It records the execution trace of a multi-agent pipeline run; if an output of that pipeline is a media artefact subject to Art. 50(2), the protocol's `output_hash` field can link to the artefact, and a C2PA manifest on that artefact can carry the user-facing disclosure mark. The two standards are complementary, not overlapping.

This distinction is worth making explicit because conversations about "AI provenance" routinely conflate the two. C2PA is not a competitor to PROV-AGENT or to this protocol; it operates at a different layer of the stack.

## 2.5 The EU AI Act

Regulation (EU) 2024/1689, the Artificial Intelligence Act, entered into force on 1 August 2024 with phased application across 2025 and 2026. It is the world's first comprehensive horizontal regulation of AI systems. It defines a risk-tiered framework: unacceptable-risk practices are banned outright, high-risk systems carry the bulk of the technical obligations, limited-risk systems carry transparency obligations, and minimal-risk systems are largely unregulated.

The protocol proposed here is concerned with three articles. Articles 12 and 14 fall in the high-risk-systems chapter (Chapter III, Section 2). Article 50 is broader in scope and applies to any AI system interacting with natural persons or generating synthetic content.

A full obligation-by-obligation extraction is given in `obligation_summary.md`. The summary below covers what is most relevant for situating Chapter 3.

### 2.5.1 Article 12 — Record-Keeping

Article 12 requires that high-risk AI systems "technically allow for the automatic recording of events (logs) over the lifetime of the system." Three observations matter for the protocol design.

First, the obligation is *technical*, not procedural. The system must allow automatic recording — operators cannot satisfy Art. 12 by writing a runbook that asks engineers to remember to enable logging. This is the central justification for the middleware-as-infrastructure approach taken in Chapter 4: the logging hooks must be built into the pipeline framework itself, not bolted on at deployment time.

Second, Art. 12(2) lists three purposes the records must serve: identifying risk situations (cross-referencing Art. 79(1)), facilitating post-market monitoring (Art. 72), and enabling operational monitoring by deployers (Art. 26(5)). These purposes are met not by per-record risk flags but by the existence of a queryable record set, a stable identification of the model and its version on every record (for drift detection), and a consistent identifier for the pipeline run (for audit linkage).

Third, Art. 12(3) lists minimum record contents *for biometric identification systems specifically*: start and end times of use, the reference database consulted, the input data, and the natural persons who verified the result. The thesis treats these minima as the floor and applies them universally across all record types. This is a design choice: a protocol that meets the biometric minimum on every record is straightforwardly compliant for biometric pipelines and over-compliant (harmlessly) for others.

The obligation that maps most directly to the central novelty of this protocol is Art. 12(3)(d) — the requirement to identify the natural persons who verified the result. This is satisfied by the `reviewer_id` field on the Human Intervention Record.

### 2.5.2 Article 14 — Human Oversight

Article 14 requires that high-risk systems be "designed and developed in such a way" that natural persons can effectively oversee them during use. The five sub-clauses of Art. 14(4) enumerate what oversight persons must be enabled to do: understand the system's capabilities and limitations, remain aware of automation bias, correctly interpret the output, disregard or override the output, and intervene or halt the system through a stop mechanism.

Article 14 is, in form, primarily a *design* obligation. It mandates that the system *be designed* to permit oversight; it does not directly mandate that oversight events be logged. But in practice, design obligations require evidence, and the only credible evidence that an oversight capability exists and is exercised is a record of its exercise. A pipeline that emits a Human Intervention Record every time a human reviewer approves, rejects, edits, or escalates an agent output produces a continuous audit trail that demonstrates Art. 14 compliance not just at design review but in operation.

Within Art. 14, the sub-clause that most directly motivates this protocol is 14(4)(d): the right to disregard, override, or reverse the output. To prove that a human had the ability to override, one must record at minimum the state of the output before the human saw it (`output_before_hash`), the state after the human's decision (`output_after_hash`), and the nature of the decision (`action_type`). No existing provenance standard captures this triple. This is the gap the Human Intervention Record fills.

A note on Art. 14(5): for biometric identification systems, "no decision shall be taken unless verified and confirmed by at least two natural persons." This justifies modelling `reviewer_id` as an array (rather than a single value) on the Human Intervention Record, so that dual-sign-off can be expressed uniformly.

### 2.5.3 Article 50 — Transparency

Article 50 imposes transparency obligations on a broader set of AI systems than Articles 12 and 14. It requires that natural persons be informed when they are interacting with an AI system (50(1)), that AI-generated content be machine-readably marked (50(2)), that emotion recognition and biometric categorisation systems be disclosed to exposed persons (50(3)), and that deepfakes be disclosed to recipients (50(4)).

Here, Art. 50 is a secondary mapping. Most of its obligations are user-facing disclosure requirements that belong on the application surface, not in the provenance trace. The protocol can record *whether* disclosure was presented (a `disclosure_presented` boolean on the Pipeline Bundle), but it cannot enforce that the disclosure was correctly worded or rendered.

One Art. 50 hook is more substantive. Article 50(4) contains an exemption: AI-generated text that has undergone human review with editorial responsibility is exempt from the deepfake-disclosure obligation. The Human Intervention Record, with `action_type: approved` and a `reviewer_role` indicating editorial responsibility, constitutes the technical evidence of that exemption. This is a useful but secondary benefit of the protocol — it does not motivate the design but is enabled by it.

### 2.5.4 What the Act does not say

The Act, in the articles relevant here, does not prescribe a record format, a hashing algorithm, a serialisation, or an integrity scheme. It says that records must exist, must enable specific oversight and monitoring functions, and must identify specific actors. The choice of how to satisfy those obligations technically is left to system designers. This is the space the protocol occupies: a concrete, machine-readable schema whose fields correspond, clause by clause, to the Act's requirements.

## 2.6 Synthesis — what the gap is

The four bodies of work above leave a specific shape of gap.

PROV-O provides the vocabulary but no AI-specific or compliance-specific schema. PROV-AGENT provides an AI-specific schema for automated agent execution but nothing for human oversight and no regulatory mapping. C2PA provides artefact-level marking but nothing about pipeline execution. The EU AI Act prescribes obligations but no protocol to satisfy them.

A complete provenance scheme for a regulated multi-agent pipeline therefore needs four things that no single existing standard provides:

1. **A record type for human oversight events**, modelling at minimum the reviewer's identity, the action taken, the output state before the decision, the output state after, and the timestamp of the intervention. PROV-AGENT does not provide this. PROV-DfA models human action but not human oversight of AI output. The EU AI Act requires the capability but does not specify the schema.

2. **An explicit field-to-clause mapping** between the protocol's record contents and the obligations of EU AI Act Articles 12, 14, and 50. No existing AI provenance standard offers this; PROV-AGENT contains no references to the Act or to any other regulatory framework.

3. **A privacy-respecting default storage model**, in which prompt and response *content* remains in application storage and the provenance record stores cryptographic hashes (`input_hash`, `output_hash`) rather than raw text. PROV-AGENT permits but does not default to this; for a deployment subject to GDPR data-minimisation principles, storing raw LLM inputs in the provenance record is often unacceptable.

4. **A pipeline-level integrity seal**, in which the entire ordered set of records for a pipeline run is hashed (`bundle_hash` over canonical JSON) so that post-hoc tampering with the record set is detectable. PROV-AGENT has no equivalent.

Chapter 3 specifies the four record types — Agent Step, Tool Invocation, Human Intervention, Pipeline Bundle — that together discharge these four requirements, and walks through the field-to-clause mapping for each.
